### 2.1 A1. 痛点提炼与用户画像

#### 2.1.1 用户画像与核心诉求

| 角色 | 代表反馈 | 主要特征 | 核心诉求 |
| --- | --- | --- | --- |
| 老年患者 | 王大爷 | 视力一般、对高密度信息不友好、对过号成本敏感 | 看得清当前号码、听得见提醒、过号后不至于完全失控 |
| 外籍患者 | David Smith | 不懂中文、需要独立完成候诊 | 看得懂英文界面、听得懂英文播报、少依赖他人帮助 |
| 特殊紧急患者 | 李女士 | 症状明显、当前系统无法动态调序 | 可以发起紧急优先申请，并尽快被医生注意到 |
| 医生 | 赵主任 | 需要稳定问诊节奏、厌恶被频繁打断 | 队列清晰、叫号可控、患者不要反复敲门追问 |
| 现场工作人员/护士 | 张姐 | 真实承担秩序维护，但本轮不建设独立系统端 | 在断网和争议场景下有清晰的人工兜底方式 |

#### 2.1.2 从反馈到问题归类

| 反馈来源 | 原始问题 | 归类后的产品问题 |
| --- | --- | --- |
| 王大爷 | 大屏太密、广播听不清、过号后等待过长 | 公共信息可读性差、提醒通道单一、过号体验差 |
| David Smith | 屏幕和播报都无法理解 | 系统缺少英语支持 |
| 李女士 | 紧急患者无法通过系统被优先处理 | 队列缺少动态优先机制 |
| 赵主任 | 患者不断打断问诊节奏 | 队列透明度低、医生端缺少有效控制面板 |
| 张姐 | 插队难解释、真假紧急难判断、断网后系统瘫痪 | 缺少优先申请判定机制、缺少断网降级方案 |

#### 2.1.3 归纳出的核心功能需求

1. 必须有一个实时同步的统一队列，让三端看到的是同一状态。
2. 必须把“当前叫号”做成最强视觉信息，大屏端能被远距离识别。
3. 必须支持中英文界面，至少让外籍患者能理解关键状态。
4. 患者端必须能看到“我的号码、前方人数、预计等待时间、诊室号”。
5. 医生端必须支持“呼叫下一位、跳过当前、暂停/恢复叫号”。
6. 患者端必须提供“紧急优先申请”入口。
7. 紧急优先申请不能自动插队，而是先经 AI 分析，再由医生决定批准或驳回。
8. 断网时不能假装正常运行，必须显式展示“当前断网”和“断网前最新状态”。

#### 2.1.4 本轮 MVP 的建模结论

基于题目限时交付约束，本轮系统只建设：

- 医生端 Web
- 候诊大屏端
- 患者端 H5
- 后端 API + WebSocket 服务

虽然护士长的反馈暴露了真实业务痛点，但本轮不单独建设护士端；相关问题在 MVP 中被折叠为：

- 医生端审核紧急优先申请
- 大屏/患者端在断网时提示“听从现场工作人员或护士指引”

### 2.2 A2. 功能优先级矩阵

#### 2.2.1 P0：MVP 必须完成

| 功能 | 说明 | 分级理由 |
| --- | --- | --- |
| 队列实时同步 | 医生端、大屏端、患者端状态一致 | 这是叫号系统成立的前提，没有同步就没有多端系统 |
| 医生端叫号/跳过/暂停/恢复 | 医生推进当前诊室的叫号流程 | 直接决定主流程能不能跑通 |
| 患者端个人排队信息 | 展示号码、前方人数、预计等待时间、诊室号 | 直接缓解焦虑与重复询问，是患者端存在的核心价值 |
| 候诊大屏高可读展示 | 大字号、高对比、当前叫号高亮 | 直接回应王大爷“看不清”的痛点 |
| 中英文界面 | 关键页面与状态支持英文理解 | 直接回应 David 的需求，且改造成本相对可控 |
| 页面刷新后状态恢复 | 任意端刷新后恢复当前正确状态 | 保证状态一致性，符合题目技术要求 |

#### 2.2.2 P1：重要但可延后

| 功能 | 说明 | 分级理由 |
| --- | --- | --- |
| 紧急优先申请入口 | 患者可提交一句话描述 | 没有这个入口，李女士类场景仍然只能靠线下喊人 |
| 医生人工审核优先申请 | 医生基于 AI 结果决定批准/不批准 | 本轮不建设护士端，审批权必须明确落在医生端 |
| AI 结构化评估能力 | 为优先申请提供辅助判断 | 很重要，但本质上是“优先申请”的增强能力，而不是独立主功能 |
| 断网检测与离线降级展示 | 断网时显式提示并保留断网前最新状态 | 对安全很重要，但在最小演示闭环中可晚于主叫号流程落地 |
| 过号重返队列策略 | 患者过号后如何重新回到队列 | 真实高频，但规则复杂，可先人工处理再产品化 |

#### 2.2.3 P2：锦上添花

| 功能 | 说明 | 分级理由 |
| --- | --- | --- |
| 双语语音播报 | 中文后英文播报当前叫号 | 有价值，但低于核心队列同步和界面可读性 |
| 更细粒度隐私模式 | 更多姓名脱敏与展示策略 | 当前固定脱敏规则已能满足 MVP |
| 预计等待时间智能预测 | 基于更复杂模型动态估算 | MVP 先用简单规则估算即可 |
| 患者满意度收集 | 候诊结束后反馈 | 不影响核心业务闭环 |

#### 2.2.4 分级原则总结

本项目的优先级判断标准不是“功能听起来是否高级”，而是看它是否直接影响以下三个核心闭环：

1. 能不能稳定完成一次叫号
2. 患者能不能明确知道自己排到哪里
3. 医生能不能在被少打断的情况下推进队列

凡是直接支撑这三个闭环的能力，进入 `P0`；重要但可以先靠简化方案或人工兜底的，进入 `P1`；体验增强型能力，进入 `P2`。

### 2.3 A3. 离线降级方案

本方案专门针对反馈 #5 中“内网交换机故障 40 分钟导致系统瘫痪”的问题设计。本轮不追求断网时继续复杂自动写操作，而是优先保证：

- 各端第一时间知道系统已断网
- 各端继续展示断网前最后一版有效状态
- 患者能获得统一、稳定、双语的安抚与引导信息

#### 2.3.1 断网检测机制

系统通过以下三类信号判断是否进入断网状态：

1. WebSocket 心跳超时
2. WebSocket 连接断开且多次自动重连失败
3. 获取最新队列快照的接口连续失败

判定原则：

- 满足任一关键条件即可切换到断网展示模式
- 前端不继续假设“只是短暂卡顿”，而是显式进入异常态

#### 2.3.2 降级运行策略

##### 医生端

- 页面顶部以高对比 Banner 显示“当前网络异常”
- 继续保留断网前最后一次成功同步的队列快照
- 明确提示“当前数据可能不是最新，请结合现场情况判断”

##### 候诊大屏端

- 保留断网前最后一版有效队列状态
- 显示显眼的断网标识
- 循环显示双语公告：
  - 中文：`当前系统网络故障，请不要紧张，听从现场工作人员或护士指引`
  - 英文：`The queue system is temporarily offline. Please stay calm and follow the instructions of on-site staff or nurses.`

##### 患者端

- 显示“网络连接异常”提示
- 保留断网前最后一次个人候诊状态
- 提示患者不要仅依赖手机数据，需留意现场工作人员指引

#### 2.3.3 恢复后数据同步策略

网络恢复后采用“重新对齐快照”的策略，而不是复杂的离线补写：

1. 客户端自动重连 WebSocket
2. 重连成功后立即重新拉取完整队列快照
3. 用服务端最新快照覆盖断网期间保留的旧状态
4. 各端重新进入正常同步模式

这样做的原因是：

- 当前 MVP 不建设复杂离线写入和冲突合并机制
- 在医疗场景里，错误同步比延迟恢复更危险
- 对于限时题目，优先保证“状态可见、恢复清楚”比“离线强行可写”更稳妥

#### 2.3.4 方案取舍说明

本轮离线降级是“展示降级”而不是“业务自治降级”：

- 不追求断网期间系统继续自动维护完整队列逻辑
- 不实现复杂的本地操作日志和回放机制
- 把断网期间的现场秩序维护交还给工作人员/护士

这个取舍符合题目限时、MVP 交付和医疗安全三者之间的平衡。

### 2.4 A4. AI 使用说明

#### 2.4.1 本项目中使用 AI 的环节

本项目在以下环节使用了 AI 工具辅助：

1. 原始反馈归纳：把 5 条口语化反馈归纳为结构化用户痛点、角色诉求和功能需求
2. PRD 初稿生成：先形成业务对象、流程、优先级和异常场景的初始框架
3. 文案整理：帮助生成 README、PRD、设计文档和 API 架构文档的初稿
4. AI 能力设计：设计“紧急优先申请”的 Prompt 结构、输出 JSON 格式与反滥用规则

#### 2.4.2 AI 与人工判断的分工

AI 负责：

- 快速整理原始信息
- 生成结构化初稿
- 提供多种需求拆解与表述方式
- 辅助补充边界情况，例如断网提示、英文支持、状态建模

人工负责：

- 决定哪些内容保留，哪些内容删除
- 收紧 MVP 范围，避免过度设计
- 重新指定角色边界，例如本轮不建设护士端、审批权改为医生端
- 重排功能优先级，例如把双语语音播报降到 `P2`
- 校正 AI 容易过度扩张的问题，使方案真正贴合题目限时交付要求

#### 2.4.3 为什么不能直接照搬 AI 结果

原始题目明确提醒不能“复述式使用 AI”。如果直接照搬 AI 初稿，最容易出现的问题是：

- 角色越写越多，超出题目交付边界
- 功能越堆越全，脱离 MVP
- 离线方案和审计方案写得过重，但实现上不现实

因此本项目采用的是：

- `AI 初稿`
- `人工 Review`
- `人工删改收敛`

最终保留的结论以人工判断为准，而不是以 AI 输出为准。

#### 2.4.4 本项目中的 AI 使用结论

AI 在本项目中更适合作为“需求整理与方案发散工具”，而不适合作为最终产品判断者。  
真正决定交付质量的关键，仍然是人工是否根据题目约束做了明确取舍。

#### 2.5.1 产品定位与边界

MediQueue 的产品定位不是普通“门诊叫号屏”，而是一套围绕门诊候诊场景设计的多端实时协同系统。它的核心价值不是“显示号码”，而是同时解决以下四个问题：

1. 患者是否知道现在叫到谁、自己还有多久轮到
2. 医生是否能在少被打断的情况下稳定推进问诊
3. 特殊患者是否能通过系统发起优先申请，而不是只能靠现场争取
4. 断网时系统是否还能稳定传达状态，而不是直接失控

结合题目限时要求，本轮边界明确为：

- 医生端 Web
- 候诊大屏端
- 患者端 H5
- 后端 API + WebSocket 服务

本轮不扩展：

- 护士独立端
- 院方管理后台
- 审计系统
- 多院区/多科室统一调度

#### 2.5.2 用户角色与核心场景建模

虽然系统最终交付是 `3 端 + 1 后端`，但业务建模必须覆盖真实利益相关方：

| 角色 | 关键场景 | 系统响应方式 |
| --- | --- | --- |
| 老年患者 | 看不清、听不见、容易过号 | 大屏大字号高对比、当前叫号高亮、患者端个人状态明确 |
| 外籍患者 | 无法理解中文屏幕和广播 | 中英文界面、关键提示英文可理解、双语播报预留 |
| 特殊患者 | 症状紧急但无法被系统优先处理 | 患者端发起紧急优先申请，AI 分析后医生审批 |
| 医生 | 被频繁打断、队列不透明 | 医生端统一控制台、呼叫/跳过/暂停/恢复能力 |
| 现场工作人员 | 断网或争议场景需要现场维持秩序 | 系统在断网时显式引导患者听从现场指引 |

核心场景可抽象为 5 条主链路：

1. 普通患者进入队列并等待叫号
2. 医生推进队列并广播当前叫号
3. 患者端收到状态更新与被叫号提醒
4. 特殊患者发起优先申请并由医生决策
5. 网络异常时三端进入断网展示模式

#### 2.5.3 业务对象与状态设计

从实现最小闭环出发，当前方案只保留 5 个核心业务对象：

| 对象 | 作用 |
| --- | --- |
| `Patient` | 存储患者身份、姓名、性别、语言偏好等基础信息 |
| `QueueTicket` | 表示一次候诊主体，承载号码、状态、诊室、优先级 |
| `PriorityRequest` | 表示一次紧急优先申请 |
| `CallEvent` | 表示一次叫号事件 |
| `QueueSnapshot` | 表示任一时刻完整的队列快照 |

队列状态设计为：

- `WAITING`
- `CALLED`
- `SKIPPED`
- `IN_CONSULTATION`
- `COMPLETED`
- `MISSED`

候诊排序规则设计为：

- `NORMAL`
- `PRIORITY_REVIEWING`
- `PRIORITY_APPROVED`
- `RETURNING`

这套模型的目的是把两个问题拆开：

- “这个人当前处于什么阶段”由 `状态模型` 解决
- “多个等待中的人谁更靠前”由 `候诊排序规则` 解决

#### 2.5.4 核心业务规则

完整方案中的关键规则如下：

1. 患者提交紧急优先申请后，不自动插队。
2. AI 只提供分析结果，不提供最终医疗决策。
3. 医生端是当前 MVP 中唯一的优先审批入口。
4. 非医疗理由如“VIP”“赶时间”必须被识别为无效优先理由。
5. 医生只能操作所属诊室队列。
6. 大屏端是公共终端，因此姓名必须脱敏。
7. 断网时系统优先表达“当前异常状态”和“断网前最新状态”，不追求复杂离线自治。

#### 2.5.5 三端功能方案

##### 医生端 Web

医生端是“控制台”，其目标是减少问诊被打断并稳定推进队列。

核心功能：

- 查看当前队列
- 查看当前叫号对象
- 呼叫下一位
- 跳过当前
- 暂停叫号 / 恢复叫号
- 审核紧急优先申请

设计重点：

- 当前叫号卡片必须高亮
- 主操作层级必须非常明确
- 紧急优先申请应以弹窗或显著提醒进入医生视线

##### 候诊大屏端

大屏端是“广播屏”，目标是让远距离、嘈杂环境中的患者一眼理解当前状态。

核心功能：

- 展示当前叫号号码
- 展示当前患者脱敏姓名
- 展示诊室号
- 展示前 5 位等待队列
- 展示正常 / 暂停 / 断网状态
- 如实现则触发双语语音播报

设计重点：

- 大字号
- 高对比
- 当前号码绝对主焦点
- 断网时切换为状态公告优先

##### 患者端 H5

患者端是“个人候诊视图”，目标是降低焦虑，而不是展示全局复杂队列。

核心功能：

- 我的号码
- 当前状态
- 前方人数
- 预计等待时间
- 诊室号码
- 紧急优先申请入口
- 被叫号时的视觉提醒与震动提醒

设计重点：

- 我的号码是页面第一视觉中心
- 信息层级少而明确
- 优先申请输入尽量简单，一句话即可

#### 2.5.6 UI/UX 风格设计分析

完整视觉方案参考 [design.md](./docs/design.md)，整体风格关键词是：

- Calm
- Precise
- High Contrast
- Signal-driven

设计主题定义为 `Clinical Signal System`，本质上是一套“医疗场景下的信号系统”。

视觉系统关键结论：

1. 颜色采用“深蓝墨色 + 医疗青绿 + 琥珀提醒 + 急诊红”。
2. 数字优先于文字，号码、前方人数、诊室号必须是最高层信息。
3. 大屏和医生端偏深色控制台风格，患者端偏浅色陪伴式卡片风格。
4. 断网、当前叫号、优先风险等状态都要用显著颜色和组件表达。

三端风格差异：

| 端 | 风格定位 | 视觉重点 |
| --- | --- | --- |
| 医生端 | Clinical Console | 稳定控制、操作清晰、状态集中 |
| 大屏端 | Broadcast Signal Board | 当前号码绝对主焦点、远距可读 |
| 患者端 | Calm Companion | 焦虑缓释、我的状态优先 |

组件层建议统一抽象：

- `StatusChip`
- `CurrentCallHero`
- `OfflineBanner`
- `PriorityReviewModal`

这样可以保证三端既统一语义，又保留各自的展示气质。

#### 2.5.7 后端架构设计分析

完整后端架构参考 [backend_api_architecture.md](./docs/backend_api_architecture.md)。当前后端技术栈定义为 `Python + FastAPI`。  
当前 MVP 后端可抽象为 5 个核心模块：

| 模块 | 作用 |
| --- | --- |
| API Layer | 提供 REST 接口、参数校验、响应封装 |
| WebSocket Gateway | 维护连接、广播状态变更、支持重连同步 |
| Queue Service | 管理主队列状态与叫号逻辑 |
| Priority Review Service | 处理优先申请、AI 结果与医生审批 |
| LLM Adapter | 封装真实 LLM 调用或 Mock 调用 |

当前状态存储采用 `In-Memory Queue Store`，理由是：

- 题目明确允许使用 mock 数据
- 先验证产品逻辑比先建数据库更重要
- 内存模型足够支撑 MVP 的实时演示闭环

#### 2.5.8 预期 REST API 设计

后端最小 REST API 集合如下：

| 方法 | 路径 | 作用 |
| --- | --- | --- |
| `GET` | `/api/v1/queue` | 获取完整队列快照 |
| `GET` | `/api/v1/patients/{patient_id}/queue-view` | 获取患者个人视图 |
| `POST` | `/api/v1/calls/next` | 医生呼叫下一位 |
| `POST` | `/api/v1/calls/skip` | 医生跳过当前 |
| `POST` | `/api/v1/calls/pause` | 暂停叫号 |
| `POST` | `/api/v1/calls/resume` | 恢复叫号 |
| `POST` | `/api/v1/priority-requests` | 提交紧急优先申请 |
| `POST` | `/api/v1/priority-requests/{request_id}/review` | 医生审核优先申请 |
| `GET` | `/api/v1/health` | 健康检查与断网探活 |

WebSocket 事件最小集合如下：

- `snapshot.sync`
- `queue.updated`
- `call.started`
- `call.skipped`
- `call.paused`
- `call.resumed`
- `priority.reviewed`

#### 2.5.9 AI 与实时架构的结合方式

这个系统的技术特色不是单纯“上了 AI”，而是把 AI 放在一个不会破坏主流程的位置：

1. 患者发起一句话优先申请
2. 后端调用 LLM 或 Mock Adapter
3. 返回结构化 JSON
4. 医生端弹出审核提示
5. 医生批准或驳回
6. 队列更新后通过 WebSocket 广播到三端

也就是说：

- AI 只参与“建议生成”
- 实时系统负责“状态传播”
- 医生负责“最终决策”

这种链路既保留了 AI 的价值，也避免了把医疗优先判断完全交给模型。

#### 2.5.10 非功能需求与风险

完整方案除了功能闭环，还必须满足以下非功能要求：

- 实时性：医生操作后 1 秒内三端同步
- 一致性：刷新后恢复到最新快照
- 可用性：WebSocket 自动重连
- 无障碍：大字号、高对比、双语、视觉 + 震动提醒
- 隐私：大屏与公共信息默认脱敏

主要风险包括：

- AI 判断不稳定
- 浏览器 TTS 在不同设备上表现不一致
- 预计等待时间受现实波动影响较大
- 断网期间展示的是旧状态，需要现场人工兜底

#### 2.5.11 发布与实现建议

从交付顺序上，建议优先实现：

1. 内存队列 + `GET /queue`
2. 医生端叫号 / 跳过 / 暂停 / 恢复
3. WebSocket 广播与重连恢复
4. 患者端个人视图与大屏端广播视图
5. 紧急优先申请 + AI 结构化输出 + 医生审核
6. 最后再补双语语音播报和更细节的体验增强

从发布策略上，建议按“单诊室、单候诊区”进行受控验证，先确保：

- 多端实时同步稳定
- 当前叫号与前方人数表达清楚
- 医生审批优先申请闭环跑通
- 断网展示逻辑足够清晰

#### 2.5.12 PRD 分析结论

综合来看，MediQueue 的最小正确解不是“做一个更好看的叫号屏”，而是构建一个：

- 医生可控
- 患者可理解
- 外籍患者可读
- 特殊患者可申请
- 网络异常时可感知

的多端实时候诊系统。

这也是为什么本项目的完整 PRD 分析会同时覆盖：

- 业务问题建模
- 功能优先级
- 三端设计风格
- 后端 API 与实时架构

因为只有这几部分一起成立，MVP 才算真正闭环。
