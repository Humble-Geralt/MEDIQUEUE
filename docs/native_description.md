二、业务场景
你刚加入一家医疗科技公司，接手某三甲医院运行 5 年的叫号系统。该系统每天服务约 800 名患者，目前处于严重失控状态。以下是你从不同角色收集到的 原始反馈（这是你唯一的需求来源，没有 PRD）：

📋 用户反馈原始记录
【反馈 #1】患者 · 王大爷（72岁，轻度老花）

"大屏幕上密密麻麻全是字，我根本看不清哪个是我的号。走廊里人多吵得很，喊我名字我也没听见，结果过号了。去分诊台重新排，护士说要等前面 6 个人看完。我都站了快两小时了。"

【反馈 #2】患者 · David Smith（外籍教师，不懂中文）

"I couldn't understand anything on the screen or the announcements. I missed my turn twice. My colleague had to come help me. The hospital really needs English support."

【反馈 #3】患者 · 李女士（孕 36 周，腹痛）

"我肚子疼得受不了，但前面还有 10 个人，他们好多就是来开降压药的常规复诊。护士让我等着，说系统里没法调顺序。后来是医生亲自出来把我带进去的。"

【反馈 #4】医生 · 赵主任（心内科）

"我的问诊节奏完全被打乱了。一会儿有人推门进来问到没到他，一会儿有人说'就看一眼化验单一分钟就走'。这些人其实都不在队列里。我需要一个清晰的队列，让患者自己知道排到哪了，别再来敲门了。"

【反馈 #5】护士长 · 张姐（分诊台负责人）

"最大的问题有三个：① 我手动给急症患者提前了，外面排队的人不知道为什么被插队，直接在走廊吵架；② 有些患者明明不急，也嚷嚷要插队，我很难判断谁是真的紧急；③ 上个月内网交换机故障了 40 分钟，叫号系统彻底瘫了，现场差点出医疗事故。"

三、任务要求
请基于以上 5 条反馈，完成以下三个交付物。

📦 交付物 A：业务分析文档（写在 README.md 中）
本部分占总分 30%，请认真对待。

必须包含以下内容：  
章节	要求
A1. 痛点提炼与用户画像	从 5 条反馈中提炼出不同角色的核心诉求，归纳为具体的功能需求
A2. 功能优先级矩阵	将所有功能分为 P0（MVP 必须）/ P1（重要但可延后）/ P2（锦上添花），并写出分级理由
A3. 离线降级方案	针对反馈 #5 中的断网场景，设计一套离线降级方案（文字描述优先，不要求必须真实实现代码），要求覆盖：断网检测机制、降级运行策略、恢复后数据同步策略
A4. AI 使用说明	简述你在哪些环节使用了 AI 工具、如何使用、AI 生成内容与你的人工判断如何分工

📦 交付物 B：多端实时叫号系统（代码实现）
本部分占总分 30%，考察实时架构与多端适配能力。

实现以下三个端 + 一个后端服务：

B1. 后端服务（API + WebSocket Server）
•维护一个内存中的患者队列（初始化 10 个 mock 患者，含姓名/号码/科室/状态等）
•提供 REST API：获取队列状态、提交插队申请等
•提供 WebSocket 通道：推送叫号事件、队列变更事件

B2. 医生端（Web PC 界面）
•显示当前等候队列
•操作按钮："呼叫下一位"、"跳过当前"、"暂停叫号"
•点击"呼叫下一位"后，通过 WebSocket 通知所有端

B3. 候诊大屏端（TV 全屏模式）
•模拟候诊区电视屏设计，适合远距离观看
•要求：大字体（建议 ≥ 48px）、高对比度配色、当前呼叫号码醒目高亮
•显示内容：当前呼叫号码+姓名+诊室 / 等候队列前 5 位
•触发语音播报（详见交付物 C2）

B4. 患者端（移动 H5 模拟）
•以某一位患者视角展示：我的号码、前面还有几人、预估等待时间
•包含"申请紧急优先"入口（详见交付物 C1）
•当被叫号时，前端出现明显的视觉+震动提示

核心技术要求
•医生点击"呼叫下一位" → 大屏端与患者端 1 秒内无刷新同步更新
•WebSocket 断线自动重连机制
•状态一致性：任何端刷新后能恢复到当前正确状态

📦 交付物 C：AI 能力集成（代码实现）
本部分占总分 20%，考察 LLM Prompt Engineering 与 AI 能力落地。

C1. AI 紧急度评估（LLM 集成）
场景：患者在 H5 端提交"申请紧急优先"，需要输入一句话描述症状/理由。

要求实现：

1.后端接收申请文本后，调用 LLM API 进行紧急度分析
2.LLM 必须返回 结构化 JSON

Prompt 鲁棒性要求：

•必须能抵御"我是VIP"、"我赶时间"等非医疗理由的滥用
•必须能识别"胸闷气短"、"大量出血"等真正紧急情况

如果你没有可用的 LLM API Key：请在代码中保留完整的 Prompt 设计和 API 调用逻辑，用 Mock 函数返回预设结果，并在 README 中标注。

C2. 双语语音叫号（TTS 集成）
场景：每次医生叫号成功后，大屏端需触发语音播报。

要求实现：

1.先播中文，再播英文（间隔 0.5 秒）
2.播报模板：

•中文："请 {号码} 号 {姓名}，到 {诊室} 就诊"
•英文："Number {号码}, {英文名/拼音}, please proceed to {诊室}"

如果没有付费的api key，可尽量寻找开源方案或者免费配额做完本地作答调试之后，在README中标注方案，无需提交key
