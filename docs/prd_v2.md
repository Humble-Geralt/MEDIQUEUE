## 2. 产品背景

### 2.1 业务背景

医院现有叫号系统已运行 5 年，每日服务约 800 名患者，但在高噪音、高人流、弱网络、角色诉求冲突的真实门诊环境中，已出现明显失控：

- 老年患者看不清屏幕、听不清播报，容易过号
- 外籍患者无法通过中文界面独立完成候诊
- 特殊紧急患者无法通过系统获得优先处理
- 医生频繁被患者打断，问诊节奏被破坏
- 护士缺乏统一依据来处理插队和争议
- 网络中断后系统整体瘫痪，存在医疗安全风险

### 2.2 核心问题

MediQueue 要解决的不是单一“叫号显示”问题，而是一个多角色协同问题：

1. 患者要知道“什么时候轮到我”
2. 医生要稳定推进“下一位是谁”
3. 护士要能够“合理插队且说得清楚”
4. 医院要确保“断网时也不能乱”

### 2.3 目标用户

- 患者：普通患者、老年患者、外籍患者、特殊紧急患者
- 医生：门诊医生、专家门诊医生
- 分诊人员：护士长、分诊台护士
- 管理方：门诊运营/信息科

## 3. 产品目标与非目标

### 3.1 产品目标

- 让患者明确感知自己的排队位置与当前进度，减少过号和重复询问
- 让医生通过一个可信队列推进就诊，减少无效打断
- 让分诊人员具备可解释的优先处理能力，降低现场纠纷
- 让系统在网络异常时仍可安全降级运行
- 在一个 MVP 中完成 3 端 + 1 后端 + AI/TTS 能力的最小闭环验证

### 3.2 非目标

以下内容不纳入本轮 MVP：

- 深度对接 HIS/EMR/挂号收费系统
- 完整急诊分诊体系
- 多院区统一调度
- 门诊运营 BI、大屏管理后台
- 除中英之外的多语种国际化

## 4. MVP 范围定义

### 4.1 本轮必须交付

本轮 MVP 对应题目要求的 3 个前端和 1 个后端：

- 医生端 Web
- 候诊大屏端
- 患者端 H5
- 后端 API + WebSocket 服务

同时，以下能力虽然未必以“独立端”交付，但必须在业务上被支持：

- 分诊人员审核紧急优先申请
- 记录队列调整原因
- 断网降级与恢复同步

### 4.2 范围内能力

- 实时队列同步
- 当前叫号展示
- 患者个人排队视图
- 紧急优先申请
- AI 紧急度评估
- 双语语音播报
- 队列调整留痕
- 页面刷新后状态恢复

### 4.3 范围外能力

- 短信通知
- 纸质票据打印
- 统计运营后台
- 医生排班系统
- 检验/检查/缴费等其他队列

## 6. 核心业务建模

### 6.1 业务对象

| 对象 | 含义 | 关键字段 |
| --- | --- | --- |
| Patient | 患者身份信息 | `patientId`、`name`、`englishNameOrPinyin`、`age`、`language` |
| Visit | 单次门诊就诊记录 | `visitId`、`department`、`room`、`visitType` |
| QueueTicket | 本次候诊票号 | `ticketNo`、`position`、`status`、`priorityLevel`、`checkInTime` |
| PriorityRequest | 紧急优先申请 | `requestId`、`reasonText`、`aiResult`、`reviewStatus`、`reviewNote` |
| CallEvent | 叫号事件 | `eventId`、`ticketNo`、`room`、`calledAt`、`operator` |
| QueueSnapshot | 队列快照 | `snapshotVersion`、`currentCall`、`waitingList`、`paused` |
| AuditLog | 操作审计记录 | `operator`、`action`、`before`、`after`、`timestamp` |

### 6.2 票号状态模型

- `WAITING`：等待中
- `CALLED`：已被叫号，等待到诊室
- `IN_CONSULTATION`：已接诊
- `SKIPPED`：本轮跳过
- `MISSED`：判定过号
- `COMPLETED`：本次完成
- `PAUSED`：全局暂停中

### 6.3 队列优先级模型

MVP 采用简单、可解释的优先级模型：

- `NORMAL`：普通排队
- `PRIORITY_REVIEWING`：已提交优先申请，待审核
- `PRIORITY_APPROVED`：已审核通过，可前移
- `RETURNING`：过号后重返队列

系统排序依据：

1. 分诊确认后的优先级
2. 原始排队时间
3. 当前状态是否仍在有效候诊中

### 6.4 核心业务规则

1. 患者提交紧急优先申请后，只能获得“待审核”状态，不能自动插队。
2. AI 返回的是辅助判断，不是医学结论，也不是最终排序指令。
3. 任何插队、重排、跳过操作都必须记录原因。
4. 患者端和大屏端只展示“队列已调整/有优先患者”这类简化提示，不展示病情细节。
5. 医生只能推进自己诊室队列，不能直接做人工插队审批。
6. 队列刷新后必须以服务端快照为准，不允许客户端本地状态覆盖服务端。
7. 网络异常时允许降级继续叫号，但所有离线操作必须可追溯。

## 7. 关键用户旅程

### 7.1 普通患者旅程

1. 患者进入候诊队列
2. 在 H5 查看“我的号码、前方人数、预计等待时间”
3. 候诊大屏同步展示当前叫号及队列前排
4. 医生点击“呼叫下一位”
5. 患者端收到明显提醒，大屏端高亮并播报
6. 患者前往诊室就诊

### 7.2 老年患者旅程

1. 通过大字号大屏确认当前号码
2. 通过语音播报再次确认
3. 即使错过视觉信息，也可以依赖高对比高亮和声音提示

### 7.3 外籍患者旅程

1. H5 端默认支持英文文案切换或按语言偏好展示
2. 大屏端关键字段提供英文辅助信息
3. 每次叫号时同步进行英文播报

### 7.4 紧急优先患者旅程

1. 患者在 H5 填写一句话症状/理由
2. 后端调用 LLM 返回结构化紧急度结果
3. 分诊人员查看 AI 结果并决定是否通过
4. 若通过，系统更新队列并广播简化解释
5. 若驳回，患者端展示“未满足优先条件，请按现场指引候诊”

### 7.5 网络中断旅程

1. 终端检测到与后端连接异常
2. 页面进入“离线模式”提示
3. 医护基于最近一次队列快照和现场人工指引继续最低限度叫号
4. 网络恢复后，系统拉取最新快照并补同步离线记录

## 8. 功能需求

### 8.1 后端服务

#### 8.1.1 队列服务

需求：

- 初始化 10 个 mock 患者
- 按诊室维护内存队列
- 支持获取队列快照、患者个人状态、当前叫号状态
- 支持叫号、跳过、暂停/恢复、完成就诊

验收：

- 任意时刻可返回当前标准化队列状态
- 非法状态转换应被拒绝，例如 `COMPLETED -> WAITING`

#### 8.1.2 WebSocket 实时广播

需求：

- 广播 `queue.updated`
- 广播 `call.started`
- 广播 `call.skipped`
- 广播 `queue.paused`
- 广播 `priority.reviewed`
- 广播 `system.offline` / `system.restored`

验收：

- 医生端发起叫号后，患者端和大屏端在 1 秒内收到事件
- 客户端断线重连后能恢复到最新状态

#### 8.1.3 REST API

建议最小接口集合：

- `GET /api/queue`
- `GET /api/patient/:id`
- `POST /api/call/next`
- `POST /api/call/skip`
- `POST /api/call/pause`
- `POST /api/call/resume`
- `POST /api/priority/apply`
- `POST /api/priority/:id/review`

### 8.2 医生端 Web

#### 8.2.1 页面目标

帮助医生快速识别当前队列、稳定推进看诊节奏。

#### 8.2.2 功能清单

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 查看当前叫号 | P0 | 当前号码、姓名、诊室状态 |
| 查看待叫队列 | P0 | 至少展示前若干位 |
| 呼叫下一位 | P0 | 推进队列并触发广播 |
| 跳过当前 | P0 | 处理未到场患者 |
| 暂停/恢复叫号 | P0 | 用于医生短暂中断接诊 |
| 查看特殊标记 | P1 | 优先患者、返回队列患者 |

#### 8.2.3 交互要求

- 当前叫号对象必须置顶高亮
- 暂停状态必须明显可见
- 关键操作需有反馈提示，避免重复点击

### 8.3 候诊大屏端

#### 8.3.1 页面目标

在远距离和嘈杂环境下，让患者迅速识别“现在叫到谁”和“我是否快到了”。

#### 8.3.2 功能清单

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 展示当前叫号 | P0 | 号码、姓名、诊室 |
| 展示队列前 5 位 | P0 | 低信息密度显示 |
| 当前号码高亮 | P0 | 动效或高对比突出 |
| 双语语音播报 | P0 | 中文后英文 |
| 网络/暂停提示 | P0 | 清楚说明当前状态 |
| 额外就诊引导信息 | P1 | 如请准备证件等 |

#### 8.3.3 设计要求

- 关键字号不低于 `48px`
- 高对比配色，不依赖单一颜色表达
- 避免表格化信息堆叠
- 默认全屏展示

### 8.4 患者端 H5

#### 8.4.1 页面目标

让患者以最低理解成本掌握自己的候诊状态，并能发起紧急优先申请。

#### 8.4.2 功能清单

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 我的号码 | P0 | 明确展示 |
| 前方人数 | P0 | 动态更新 |
| 预计等待时间 | P0 | 用规则估算 |
| 被叫号提醒 | P0 | 强视觉提示 + 震动 |
| 紧急优先申请 | P0 | 一句话输入 |
| 中英文界面 | P0 | 满足外籍患者需求 |
| 即将到号预提醒 | P1 | 例如前方剩 2 人 |

#### 8.4.3 业务规则

- 同一患者在短时间内不能无限提交优先申请，MVP 可采用基础限流
- 预计等待时间的 MVP 算法可为：`前方人数 × 诊室平均单人时长`
- 若浏览器不支持震动，需保留视觉高亮和弹窗提示

### 8.5 分诊审核能力

> 该能力是业务闭环所必需，但不要求本轮一定以“独立新端”呈现；可通过一个轻量审核页、管理页或带权限模块实现。

#### 8.5.1 核心功能

| 功能 | 优先级 | 说明 |
| --- | --- | --- |
| 查看待审核申请 | P0 | 看见谁在申请优先 |
| 查看 AI 评估结果 | P0 | 紧急等级、建议动作、原因 |
| 审核通过/驳回 | P0 | 最终人工决策 |
| 调整队列位置 | P0 | 通过后前移 |
| 填写调整原因 | P0 | 保证可解释和可追溯 |
| 处理过号重返 | P1 | 后续可细化策略 |

#### 8.5.2 关键要求

- 必须能解释“为什么有人被前移”
- 不能向其他患者泄露具体病情
- 审核记录必须写入审计日志

### 8.6 AI 紧急度评估

#### 8.6.1 输入

- 患者一句话症状/理由
- 可选上下文：年龄、孕周、当前科室

#### 8.6.2 输出

LLM 必须返回结构化 JSON，建议字段如下：

```json
{
  "urgencyLevel": "high|medium|low|unknown",
  "medicalReason": true,
  "isAbuseSuspected": false,
  "recommendedAction": "manual_review_priority|normal_queue|reject_non_medical",
  "explanation": "说明原因"
}
```

#### 8.6.3 规则要求

- 能识别真实医疗紧急情况，如胸闷气短、大量出血、孕晚期腹痛
- 能识别非医疗插队理由，如 VIP、赶时间、赶高铁、着急上班
- 允许模型返回 `unknown`，避免错误自动优先
- 无 API Key 时，保留完整 Prompt 和调用逻辑，并用 Mock 返回

### 8.7 双语语音播报

#### 8.7.1 播报模板

- 中文：`请 {号码} 号 {姓名}，到 {诊室} 就诊`
- 英文：`Number {号码}, {英文名或拼音}, please proceed to {诊室}`

#### 8.7.2 规则

- 中文先播，英文后播，间隔 `0.5 秒`
- 英文名缺失时回退到拼音或号码
- 播报失败不阻塞叫号主流程

## 9. 非功能需求

### 9.1 实时性

- 端到端同步时延不超过 `1 秒`
- 页面不刷新即可更新当前状态

### 9.2 一致性

- 服务端为唯一事实源
- 客户端重连或刷新后必须拉取最新快照
- 所有端对同一票号状态认知一致

### 9.3 可用性

- WebSocket 自动重连
- 重连后自动补状态
- 离线时有明确可见提示

### 9.4 无障碍与国际化

- 大字号
- 高对比
- 中文/英文双语
- 视觉提醒与声音提醒双通道并存

### 9.5 隐私与审计

- 患者端只展示与本人有关的信息
- 大屏端默认不展示详细病情
- 所有关键改序操作必须可追溯

## 10. 离线降级方案

### 10.1 触发条件

- WebSocket 心跳超时
- API 请求连续失败
- 服务端健康检查不可达

### 10.2 降级策略

#### 医生端

- 保留最后一次队列快照
- 允许进行最小叫号动作
- 所有操作进入待同步日志

#### 大屏端

- 持续展示最后一版有效队列
- 显示“当前处于离线模式，请以现场指引为准”

#### 患者端

- 提示当前状态可能延迟
- 停止展示强承诺式等待时间

#### 分诊人员

- 可通过现场人工登记或轻量本地记录继续维护顺序

### 10.3 恢复同步

- 网络恢复后先获取最新服务端快照
- 若存在本地离线操作，按时间顺序提交并校验冲突
- 冲突时以分诊人工确认结果为准
- 所有补同步操作写入审计日志

## 11. 指标与验收

### 11.1 核心业务指标

- 过号率下降
- 医生被打断次数下降
- 外籍患者独立候诊成功率提升
- 插队争议事件下降

### 11.2 核心系统指标

- 叫号广播成功率 ≥ `99%`
- WebSocket 重连成功率 ≥ `95%`
- 离线恢复后关键数据一致率 ≥ `99%`

### 11.3 验收场景

#### 场景 A：普通叫号闭环

1. 医生点击“呼叫下一位”
2. 大屏端在 1 秒内更新当前号码
3. 患者端在 1 秒内更新状态并展示提醒
4. 页面刷新后仍能看到正确状态

#### 场景 B：紧急优先申请

1. 患者提交一句话理由
2. 后端返回结构化 AI 分析结果
3. 分诊人员可查看并审核
4. 审核通过后队列顺序变更并广播
5. 其他患者仅看到“队列已调整”的简化提示

#### 场景 C：双语播报

1. 医生成功叫号
2. 大屏端先播中文，再播英文
3. 英文名缺失时可以回退处理

#### 场景 D：断网降级

1. 模拟 WebSocket 断开
2. 页面进入离线提示状态
3. 系统仍可基于最后快照维持最低限度运行
4. 恢复网络后自动同步最新状态

## 12. 风险与待确认项

### 12.1 主要风险

- AI 误判导致分诊过度依赖模型输出
- 浏览器 TTS 在不同环境中表现不一致
- 大屏展示姓名方式涉及隐私边界
- 预计等待时间容易因实际问诊波动而失准

### 12.2 待确认项

- 大屏是否允许展示全名，还是仅展示姓氏/掩码
- 过号后重返队列的具体业务规则
- 分诊审核能力在实现上是独立页面还是嵌入式模块
- 医院是否允许浏览器直接使用本地 TTS 能力

## 13. 发布建议

建议采用“单科室、单候诊区、单诊室”的受控试点方式上线：

1. 先验证多端同步、老年可读性和双语播报
2. 再验证紧急优先申请与分诊审核闭环
3. 最后验证离线降级和恢复同步策略

该发布方式可以在控制风险的前提下，尽快验证 MVP 的真实业务价值。
